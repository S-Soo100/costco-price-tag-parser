# 코스트코 가격표 인식기 — 기획서

> 작성: 2026-06-02 · 대화로 확정한 결정 + OCR 실데이터 검증 결과 통합
> 상태: 기획 확정, 빌드 착수 단계

---

## 1. 개요

코스트코 매장의 가격표를 **작업자가 현장에서 체계적으로 촬영·수집**하고, **관리자가 주 단위 변화(신규/할인/삭제)를 분석**하는 **현장 데이터 수집 + 분석 애플리케이션**.

- 플랫폼: **iOS + Android** (Flutter)
- 멀티유저 / 역할 기반: **작업자(worker)**, **관리자(admin)**
- 백엔드: **이미 존재**(별도 API). 앱은 그 API를 호출. **개발 단계에선 전부 Mock**(추후 실연동)

## 2. 개발 단계 범위 & 원칙

| 항목 | 결정 |
|------|------|
| 백엔드 | 존재하는 API 사용. **지금은 Mock 리포지토리**로 대체, 나중에 구현만 교체 |
| 매장 | 하드코딩 2개: **코스트코 양재점 / 광명점** |
| 매대 | 숫자 리스트(100·101·102…), 상태 3종: **미진행 / 진행중 / 진행완료** |
| 달성률 | 완료매대 / 총매대(서버 제공). 100매대 중 48 완료 = 48% |
| OCR | **온디바이스**, **A포맷(표준 매대카드)만** 자동 파싱. B/실패분은 관리자 수동입력 |
| 로그인 | Mock. 개발용 **역할 스위처**(작업자/관리자) |
| 금주 정의 | **달력 주(월~일) 자동** |

## 3. 아키텍처

**스택:** Flutter · Riverpod · go_router · 온디바이스 OCR(`google_mlkit_text_recognition`) · 로컬 큐(오프라인 우선)

**3개 경계 (갈아끼우기 쉽게):**
1. **데이터 = `Repository` 인터페이스 + `MockRepository`** → 나중에 `ApiRepository`로 교체, UI 불변
2. **OCR = 분리 모듈** (`OcrEngine`(플랫폼 OCR) + `PriceTagParser`(순수 Dart)). 캡처 흐름은 호출만, **실패해도 흐름 안 막힘**
3. **역할 분기 셸** — 로그인 후 작업자/관리자 셸 분기. 개발 중 역할 스위처

**오프라인 우선:** 매장 신호 약함 → 촬영·OCR은 오프라인, 업로드는 큐잉 후 온라인 시 동기화. 서버엔 **사진 + JSON** 전송·보관.

## 4. 작업자 흐름 (확정)

```
로그인(역할 스위처)
 → 매장 선택 (양재 48% / 광명 0%)
 → 매대 리스트 (100🟢 / 101🟡 / 102⚪ … + 상단 매장 달성률)
 → 매대 작업 [제품 캡처 루프]:
      [☐ 엔드캡] 체크 → 가격표 촬영(1장)
      → 제품 사진 1~5장
      → 제품 1건 저장(로컬 큐) → "다음 제품"(엔드캡 기본 해제)
      → 반복
 → 매대 완료 (확인) → 매대 🟢, 리스트 복귀, 달성률 갱신
 → 매장 완주 시 축하
```

**비동기 OCR:** 촬영 시 제품은 `pending`으로 큐 적재(대기 0) → 백그라운드가 OCR+파싱 → `auto`(성공)/`review`(실패·불확실). 캡처 흐름 안 막음.

**매대 단위 에러 표시:**
- 매대 화면/완료 시: `매대 102 · 제품 12개 · ✓10 · ⚠2`
- 매대 리스트 배지: `102 🟢완료 ⚠2`
- ⚠ 탭 → 실패 제품 목록 → **즉시 재촬영** 또는 관리자 검수로
- **"에러" 정의** = ① OCR 인식 실패(`review`) + ② 가격표 미촬영(필수 누락)
- **"매대 완료"는 OCR 처리 중이어도 막지 않음**(요약 라이브 갱신)

**잘못 찍었을 때:** 가격표 재촬영 / 제품사진 삭제 affordance 제공.

## 5. 관리자 흐름

```
로그인(관리자) → 매장 선택 + 금주(이번 주 월~일)
 → 섹션:
    · 금주 작업 현황 (달성률·진행)
    · 신규 품목   (지난주 대비 추가)
    · 신규 할인   (할인행사 시작)
    · 삭제 품목   (사라진 품목)
    · 검수·입력 대기 (B포맷·인식실패분 → 사진 보고 수동 입력)
 → 품목 상세: 촬영 사진들 · OCR 원문 · 매대 위치 · 촬영자/시각
```
- diff 단위: **(매장, 품목번호)**, 금주 vs 지난주. (서버 계산, 개발은 mock diff)
- ※ "행사(2+1)"는 코스트코에 없어 **제외**. 가격 변화 신호는 **할인행사**뿐.

## 6. 데이터 모델

**추적 키: `(storeId, itemNumber)`** — 품목번호는 전국 공통, 가격/할인은 매장별.

| 엔티티 | 필드 |
|--------|------|
| **Store** 매장 | id, name, totalRacks(서버) |
| **Rack** 매대 | id, storeId, number, status(notStarted/inProgress/done) |
| **Capture** 제품 캡처 | id, storeId, rackId, isEndcap, priceTagPhoto, productPhotos[1..5], ocrStatus(pending/auto/review/manual), priceTag(PriceTagData?), capturedAt, capturedBy |

**PriceTagData (OCR 결과, v0.1)** — 필수는 `itemNumber`·`finalPrice`·`rawText`, 나머지 optional:
```jsonc
{
  "schemaVersion": "0.1",
  "itemNumber": "685246",        // 6자리 (키)
  "plusMark": true,              // 품목번호 옆 '+' (의미 미상, raw 보관)
  "nameKo": "ROBOROCK 로보락 아쿠아청소기",
  "model": "F25 RT",
  "nameEn": "ROBOROCK AQUA VACUUM",
  "finalPrice": 349900,
  "tagType": "discount",         // regular | discount | unknown
  "discount": { "originalPrice": 389900, "discountAmount": 40000,
                "periodStart": "2026-05-21", "periodEnd": "2026-06-07" },
  "unitPrice": { "value": 11945, "unit": "개" },
  "rawText": "OCR 원문 전체",     // 항상 보관 (재파싱·검수용)
  "capturedAt": "2026-06-02T12:06:00"
}
```

## 7. OCR 파이프라인 (실데이터 검증 완료)

**검증:** 사용자 제공 실사진 46장 → macOS Vision OCR(ML Kit 동급)로 전수 검증. **35장이 A포맷(품목번호+가격), 그중 할인 6장.** 순수 Dart 파서로 추출 → 직접 확인한 표본 하드 단언 통과.

**파서 휴리스틱 (검증됨):**
| 필드 | 규칙 |
|------|------|
| 품목번호 | "6자리만 있는 줄" 우선(상단), 없으면 최상단 6자리. `+` 접미 → plusMark |
| 최종가 | **글자 높이가 가장 큰** `원`/콤마 숫자 (정가·단가는 작게 나옴) |
| 할인 | "할인행사" 라벨 존재 → discount. 정가=최종가보다 큰 값, 할인액=정가−최종가, 기간=날짜 2개 |
| 단가 | "단가 / <단위>" 라벨 옆 가까운 작은 가격 |

**실패 모드(→ 관리자 수동입력으로 흡수):** B포맷(베이커리 제품 라벨), 제품/전경샷, 흐린 태그. 같은 품목도 사진 품질 따라 파싱이 갈릴 수 있음 → 검수가 받아줌.

**연결:** 데스크톱 Vision으로 만든 파서를 그대로 사용. 디바이스에선 `OcrEngine`=ML Kit가 동일한 `{text, bbox}`를 produce → 파서 불변.

**개발 도구:**
- `sample_tags/` — 실사진 46장 (원본)
- `tools/vision_ocr.swift` — Vision OCR 프로브/fixture 생성기
- `tools/ocr_raw.json` — 46장 OCR 출력(파서 fixture)
- `tools/parser_check.dart` — 순수 Dart 파서 + 자가 검증

## 9. 미결 / 가정 (검토 바람)

- 엔드캡 체크박스 위치 = 제품 캡처 화면 상단(가격표 촬영 전), "다음 제품" 시 기본 해제 — *가정*
- `+` 기호 / 하단코드(33IOI 등) 의미 미상 → raw 보관만
- 관리자 수동입력 필드 = PriceTagData 핵심(품목번호·가격·할인) — *가정*
- 프로젝트 식별자: name `costco_price_tag_recognizer`, org `com.kangsters` — *확인 필요*
